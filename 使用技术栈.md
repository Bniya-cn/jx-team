# 使用技术栈

这份文档说明项目当前实际采用的技术栈、各层职责、选择理由，以及在这个项目里应该怎样使用它们。

---

## 1. 整体架构

JX-Team 是一个 Rails 承载、React 渲染的单页商店项目。

### 当前分层

- 后端：Rails 5.2 提供 API、认证、评论、购买、后台、翻译代理等能力
- 前端：React 17 + Redux + React Router 负责页面和状态
- 样式：SCSS 负责组件化样式与布局
- 数据库：PostgreSQL 存储用户、游戏、评测、购买、后台内容等主数据
- 构建：Webpack 5 打包前端资源
- 脚本：Node.js 脚本负责抓取 Steam 数据、做文案测试、生成静态辅助文件

### 当前数据流

1. 游戏 catalog 通过 `src/mock/games.json` 导入到 Rails 数据库
2. 前端页面通过 `/api/games`、`/api/games/:id` 等接口读取数据库数据
3. 用户登录、购买、评测、推荐、后台内容都走 Rails API
4. React 负责页面组合、交互和局部状态
5. Webpack 把前端代码打包成浏览器可用资源

---

## 2. 为什么这样选

### Rails + PostgreSQL

适合这个项目的原因：

- 用户、购买、评测这些数据天然是关系型的
- Rails 自带 MVC、路由、校验、jbuilder，适合快速搭主链路
- PostgreSQL 对关联查询、约束、事务更稳
- 后台管理和内容管理页也更适合在 Rails 上直接补

### React + Redux + Router

适合这个项目的原因：

- 商店类页面状态多，适合组件化拆分
- Redux 适合管理会话、翻译、游戏实体、购物车、愿望单等跨页面状态
- React Router 适合把首页、详情页、库、推荐、后台等页面统一起来

### Webpack 5

适合这个项目的原因：

- 能把 React、SCSS、脚本产物统一打包
- 对旧版 Rails 项目兼容性较好
- 构建命令简单，便于交接给零基础同学

### SCSS

适合这个项目的原因：

- 页面模块多，样式层级明显
- 便于把首页、详情页、后台、静态页拆成独立文件
- 适合做统一色板、间距、按钮、标签、卡片等样式管理

### Node 脚本

适合这个项目的原因：

- Steam 游戏抓取更适合用脚本批处理
- 多语言文案测试也适合脚本化
- 比手工改 JSON 更容易批量维护

## 3. 后端技术说明

### 3.1 Rails 5.2

主要职责：

- 处理用户注册、登录、退出
- 提供游戏列表、游戏详情、评测、购买、推荐、动态、统计等 API
- 提供后台管理接口
- 提供翻译代理接口

在项目里的关键文件：

- `config/routes.rb`
- `app/controllers/api/*`
- `app/models/*`
- `app/views/api/*`

### 3.2 模型设计

核心模型包括：

- `User`
- `Game`
- `Purchase`
- `Review`
- `Genre`
- `Tag`
- `NewsItem`
- `CuratorPick`

这些模型把“购买后才能评测”“一款游戏可以有多个标签”“后台内容可编辑”这类业务规则固化下来。

### 3.3 API 输出方式

项目使用 jbuilder 输出 JSON。

优点：

- 统一整理后端输出结构
- 前端拿到的就是可直接渲染的对象
- 方便集中控制游戏详情、库存、购买信息等字段

## 4. 前端技术说明

### 4.1 React 17

前端页面按业务拆分组件，例如：

- 首页和轮播
- 游戏详情
- 浏览页
- 购物车和愿望单
- 用户评测
- 推荐、动态、统计
- 后台管理

当前写法以“容器组件 + 展示组件”为主：

- 容器组件负责请求数据和状态连接
- 展示组件负责页面结构和交互

### 4.2 Redux

Redux 主要承载：

- session
- entities
- errors
- locale

适合它的原因是跨页面状态多，而且很多 UI 互相依赖同一批实体数据。

### 4.3 React Router

用于管理页面路由：

- `/`
- `/browse/:filter`
- `/api/games/:id`
- `/cart`
- `/wishlist`
- `/discover`
- `/recommendations`
- `/stats`
- `/news`
- `/curators`
- `/settings`
- `/about`

### 4.4 本地化与翻译

项目有两层语言能力：

1. 顶栏多语言切换：本地文案替换
2. 划词翻译：通过 Rails 代理 DeepSeek

对应文件：

- `frontend/util/i18n.js`
- `frontend/util/locale_strings.json`
- `frontend/components/translation/*`
- `app/controllers/api/translations_controller.rb`

如果没有配置 Key，翻译 UI 仍可工作，只是返回占位文本。

## 6. 当前项目里的实际使用方式

### 游戏数据

现在不建议前端再直接读本地 JSON。标准流程是：

1. 由 seed 把 catalog 导入数据库
2. 前端统一调 `/api/games` 和 `/api/games/:id`
3. 在组件层做本地展示和过滤

### 购物车与愿望单

- 购物车仍以浏览器本地存储为主
- 愿望单也仍以浏览器本地存储为主
- 适合演示，不适合当作长期持久化方案

### 后台管理

后台是轻量内容管理台，适合做：

- 游戏信息维护
- 标签和分类维护
- 新闻和鉴赏家配置
- 用户与评测管理

---

## 7. 选择这些技术的优缺点

### 优点

- 技术组合成熟，交接容易
- 关系型数据和业务约束好做
- 前后端边界清晰
- 适合课程项目和演示型产品

### 缺点

- Rails 版本较老
- 前端构建链偏传统
- 购物车和愿望单还没有完全服务端化
- 多语言游戏实体内容仍有补齐空间
