# 小组分工建议

这份文档用于课程作业提交时说明四位成员的职责划分，写法尽量贴近“真实项目协作”，你们可以直接按这个版本提交，也可以再按队内实际情况替换姓名。

---

## 一、分工原则

- 以前后端主线拆分，避免所有人同时改同一块代码
- 每个人都负责一块明确模块，同时保留一项联调或验收职责
- 最后一位成员承担整合、测试、文档和演示收口，保证项目能完整交付

---

## 二、四位成员分工建议

### 成员 A：前端首页与公共界面负责人

- 购买链路可演示

- 按钮、卡片、标签等细节统一

### 成员 C：后端数据与接口负责人

负责内容：

- Rails 模型、控制器、路由维护

交付目标：

- 本地可启动
- 演示流程顺畅
- 文档齐全，零基础同学也能跟着跑起来
- 最终汇报材料有清晰分工和技术说明

## 三、AI 协力与前沿工具应用（高难内容赋能）

为了应对项目中高难度的技术点与美术资源缺口，团队积极引入了主流 AI 工具作为“虚拟专家成员”，形成人机协同开发模式。具体分工与技术产出如下：

### 1. Claude：架构顾问与复杂逻辑重构
- **负责内容（高难攻关）**：
  - 设计 Rails 代理 DeepSeek API 翻译接口的后端网络请求与降级容错架构。
  - 制定从原前端本地 JSON 数据读取平滑过渡到 PostgreSQL 关系型数据库的迁移方案与 Active Record 关联模型设计（处理多对多关联、购物车与愿望单多态关联）。
  - 审查和重构遗留代码，规避 Rails 5.2 与现代前端 Webpack 构建工具的版本冲突。
- **输出成果**：保障了主数据库主链路数据结构的优雅度，编写了高健壮性的翻译代理逻辑。

### 2. Codex：代码生成与自动化测试编写
- **负责内容（高难攻关）**：
  - 协助编写复杂的 React/Redux 状态管理 Slice 与 React Router 的异步路由逻辑。
  - 快速生成符合 Rails 规范的 CRUD 控制器、路由映射以及 Active Record 复杂查询接口（如商品多分类筛选与推荐队列过滤）。
  - 批量生成 Rails 单元测试（`rails test`）与前端多语言文案校验脚本（`test:locale`），提升测试覆盖率。
- **输出成果**：缩短了样板代码的编写时间，保证了核心页面在多次迭代后的功能稳定性。

### 3. Image2 (Imagen 2)：品牌视觉系统与游戏资产生成
- **负责内容（高难攻关）**：
  - 针对无版权图片或占位图问题，使用 Image2 批量生成校园主题与游戏题材的高保真轮播图 Banner、游戏详情页的插画及游戏胶囊图（Capsule Images）。
  - 创作符合“校园游戏商店”定位的校徽结合型 LOGO、以及顶栏多语言翻译组件的精细图标，免除占位符，使系统界面极具工业设计质感。
- **输出成果**：生成了多达十余款风格统一、高清晰度的精美游戏封面资产，避免了版权纠纷，显著提升了视觉体验。

### 4. Gemini：跨平台调试、数据抓取与多语言本地化
- **负责内容（高难攻关）**：
  - 负责解决跨平台运行（如 Windows 环境下 Ruby 2.5.1 `pg` / `bcrypt` 原生 Gem 编译失败、数据库 TCP 连通性等）的高难度部署报错。
  - 编写 Node.js 脚本直接请求 Steam API 抓取真实游戏数据、分类标签与评测内容，并处理 JSON 数据清洗与后端 Seed 种子文件的高效写入。
  - 全程作为项目总控与沟通助手，统筹中英文双语包的校验与文案润色。
- **输出成果**：完成了数据爬取脚本与 Windows 部署教程的编纂，解决了 Windows 下环境编译的各种硬伤。

## 四、最终整合阶段的协作建议

为了避免最后一天冲突，建议按下面顺序整合：

1. 成员 C 先保证数据库、seed 和接口稳定，AI 助手（Claude/Gemini）协助处理迁移。
2. 成员 A 与成员 B 分头完成首页和详情页/交互页，AI 助手（Codex/Image2）提供代码和图片资产。
3. 成员 D 负责把后台、文档、演示流程补齐，AI 助手进行回归校验。
4. 四人一起完成最后一轮联调和演示彩排。

## 五、答辩时的展示话术建议

### 成员 A 可以重点讲

- 首页信息架构如何设计。
- 为什么要调整导航、布局、按钮和动效。
- 如何利用 **Image2** AI 设计校徽和校园品牌元素并融入站点，摆脱呆板占位图。

### 成员 B 可以重点讲

- 详情页为什么要补购物车、愿望单、评测和配置。
- 用户从浏览到购买再到评测的完整链路。
- 配合 **Codex** 在多语言切换与页面响应式重构中的高效率编写经验。

### 成员 C 可以重点讲

- 为什么要把数据源统一到 Rails 数据库。
- seed、接口、模型关系如何设计，以及如何由 **Claude** 辅助设计优雅的数据库 schema 迁移方案。
- 为什么关系型数据库更适合购买和评测业务。

### 成员 D 可以重点讲

- 后台管理如何支持内容维护。
- 如何使用 **Gemini** 编写数据爬虫和编写适合 Windows 运行的本地化部署教程，协助同学在没有 Mac 的情况下直接用 WSL2/原生 Windows 完成部署。
- 项目最后如何进行联调、回归和交付。
